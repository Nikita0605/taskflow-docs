# 1. Purpose

The Reporting Model defines how TaskFlow collects, organizes, and presents operational data.

It establishes the reporting architecture, data sources, reporting entities, aggregation rules, and access controls used throughout the platform.

This model provides a consistent framework for dashboards, project reports, sprint analytics, release metrics, and custom reporting capabilities.

---

# 2. Reporting Architecture

Reporting is built on data generated by platform activities.

Operational modules publish business data as work progresses. The Reporting Service collects this information, transforms it into reporting datasets, and exposes the results through dashboards and reports.

Reporting operates independently of transactional operations. Data used for reporting does not modify the source records from which it was derived.

```text
Business Activity
        │
        ▼
Reporting Service
        │
        ▼
Aggregation
        │
        ▼
Reporting Dataset
        │
        ▼
Dashboard / Report
```

---

# 3. Reporting Components

The reporting architecture consists of the following components.

| Component | Description |
|-----------|-------------|
| Data Source | Produces operational data used for reporting. |
| Reporting Service | Collects and processes reporting data. |
| Aggregation Engine | Calculates metrics and summaries. |
| Reporting Dataset | Structured data prepared for reporting. |
| Dashboard | Visual presentation of reporting information. |
| Report | Detailed presentation of reporting data. |

Each component has a distinct responsibility within the reporting architecture.

---

# 4. Data Sources

Reporting data is collected from multiple platform services.

| Source | Reporting Data |
|--------|----------------|
| Work Items | Status, priority, ownership, completion |
| Workflow | State transitions, cycle time, throughput |
| Sprint | Capacity, velocity, burndown |
| Release | Delivery history, release frequency |
| Repository | Commits, pull requests, merge activity |
| Pipeline | Build status, deployment history |

Each data source contributes information specific to its domain.

# 5. Reporting Dimensions

Reports organize information using common reporting dimensions.

| Dimension | Description |
|-----------|-------------|
| Workspace | Organizational reporting boundary. |
| Project | Project reporting boundary. |
| Team | Team-specific reporting. |
| Sprint | Iteration reporting. |
| Release | Release reporting. |
| Assignee | Individual contribution reporting. |
| Time Period | Reporting grouped by time. |

Dimensions define how reporting information is grouped and filtered.

---

# 6. Metrics Model

Metrics are calculated from reporting datasets rather than operational records.

Common metrics include:

| Metric | Description |
|--------|-------------|
| Work Items Created | Total work items created during a reporting period. |
| Work Items Completed | Total completed work items. |
| Cycle Time | Time between work start and completion. |
| Lead Time | Time from creation to completion. |
| Sprint Velocity | Work completed within a sprint. |
| Defect Count | Number of reported defects. |
| Deployment Frequency | Number of completed deployments. |

Metric definitions remain consistent across all reports.

---

# 7. Dashboard Model

A dashboard is a collection of reporting widgets.

Each dashboard presents information for a defined reporting scope.

Typical dashboard widgets include:

- Work Item Summary
- Sprint Progress
- Team Velocity
- Burndown Chart
- Release Status
- Recent Activity

Dashboards provide operational visibility without exposing raw reporting datasets.

---

# 8. Report Generation

Reports are generated from reporting datasets.

Report generation may include:

- Filtering
- Grouping
- Sorting
- Aggregation
- Time-based comparison

Report generation does not modify reporting data or operational records.

---

# 9. Access Control

Reporting follows the platform permission model.

Users may view only reporting data derived from resources they are authorized to access.

Workspace reports, project reports, and team reports are evaluated independently according to their reporting scope.

---

# 10. Reporting Constraints

| ID | Constraint |
|----|------------|
| RM-001 | Reports are generated from reporting datasets rather than operational records. |
| RM-002 | Reporting data must preserve source data integrity. |
| RM-003 | Metrics must use standardized calculation rules. |
| RM-004 | Authorization applies before reporting data is presented. |
| RM-005 | Report generation must not modify operational data. |
