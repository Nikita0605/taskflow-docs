# Project

Every product your team builds begins with a project.

Whether you're developing a customer-facing application, maintaining an internal system, or preparing the next software release, a project brings together the people, plans, and work needed to achieve a common goal.

In TaskFlow, a project acts as the working area for a single product or initiative. It keeps user stories, tasks, bugs, sprints, releases, documentation, and project reports in one place, making it easier for the team to stay organized from planning through deployment.

For example, the **Nova Hub** workspace might contain three separate projects:

- Customer Portal
- Mobile Application
- Internal HR System

Although these projects belong to the same workspace, each has its own backlog, sprint schedule, team, and release history. This allows teams to work independently without losing visibility across the organization.

---

# Create a Project

Once your workspace is ready, creating a project is the next step.

Open your workspace and select **New Project**. Enter a project name, add a short description, choose the project template that matches your team's workflow, and complete the setup.

Your new project opens immediately, ready for planning and collaboration.

> **Tip**
>
> Choose a project name that clearly represents the product or business initiative. Good names make reports, dashboards, and release histories easier to understand as the number of projects grows.

# Choosing a Project Template

Every team works differently.

Some teams follow Scrum with fixed sprint cycles, while others prefer the continuous flow of Kanban. DevOps teams may focus on deployment pipelines and operational work, whereas support teams often organize work around incoming issues.

TaskFlow provides project templates to help teams get started quickly without building a workflow from scratch.

| Template | Best For |
|----------|----------|
| Scrum | Agile teams working in sprints |
| Kanban | Continuous delivery and operational work |
| DevOps | Development, deployment, and release management |
| Blank Project | Teams that want to build a custom workflow |

The selected template creates the initial workflow. As your process evolves, you can customize boards, statuses, and project settings without creating a new project.

# Working Inside a Project

Most day-to-day work happens inside a project.

The backlog captures ideas and planned work. During sprint planning, the team selects the work they intend to complete. Developers update tasks as implementation progresses, QA engineers verify completed features, and technical writers document new functionality alongside development.

As work moves forward, dashboards and reports automatically reflect the latest project status, giving everyone a shared view of progress.

Rather than switching between different applications to understand what's happening, the team can follow the entire software delivery process from within the project.

---

# Project Settings

Project settings allow administrators and project managers to control how the project is organized.

From the settings page, you can update the project name and description, manage members, configure workflows, define release settings, and adjust notifications. Depending on your organization's configuration, additional options such as integrations, automation rules, and project permissions may also be available.

Changes made in project settings affect only the selected project and do not impact other projects within the workspace.

---

# Managing Project Members

Projects often involve people with different responsibilities. Developers build features, QA engineers verify quality, product owners manage priorities, and technical writers create documentation.

Instead of giving every member the same level of access, TaskFlow allows you to assign roles based on responsibilities. This helps protect project data while ensuring everyone has the permissions they need to perform their work.

As teams grow or responsibilities change, project membership can be updated without affecting the rest of the workspace.

---

# Project Lifecycle

A project evolves as work progresses.

It begins with planning, where objectives, requirements, and priorities are defined. The team then creates user stories, plans sprints, completes development tasks, resolves defects, and prepares releases for deployment.

Even after a release is complete, the project continues to grow through maintenance, enhancements, and future iterations. This continuous cycle allows teams to improve the product while maintaining a complete history of planning, development, testing, and delivery.

---

# Best Practices

A well-managed project is easier to maintain than one that grows without structure.

Create separate projects for separate products instead of combining unrelated work into a single project. Keep the backlog organized, review completed work regularly, and archive inactive projects when they are no longer actively maintained.

Consistent naming conventions, clearly defined responsibilities, and regular backlog refinement make projects easier to navigate as both the product and the team expand.

# Common Issues

| Issue | Resolution |
|---------|------------|
| Unable to create a project | Verify that you have permission to create projects in the workspace. |
| Team members cannot access the project | Check that they have been added to both the workspace and the project. |
| Sprint features are unavailable | Confirm that the selected project template supports sprint planning. |
| Project does not appear in search results | Verify that the project is active and that you have permission to view it. |
